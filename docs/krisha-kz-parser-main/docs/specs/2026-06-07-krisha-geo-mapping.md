# ТЗ: Справочник krisha.kz (гео-дерево + категории + схема фильтров)

**Дата:** 2026-06-07
**Статус:** на согласовании (Гейт 1)
**Связано:** память `tech_krisha_filters`, `feedback_minimize_krisha_probing`

---

## Предпосылка

Сегодня на выяснение правильного slug Бесагаша (`besagash-dzerzhinskoe`) ушли десятки
запросов к krisha (перебор slug'ов, дифф-тесты) — лишняя нагрузка и риск блокировки.
Нужен переиспользуемый справочник, чтобы будущие поиски строились по нему, БЕЗ перебора.

## Цель

Собрать и сохранить в репозитории структурированный справочник krisha.kz:
1. **Гео-дерево:** регионы → города/районы → населённые пункты/микрорайоны
   (название, URL-slug, гео-id, родитель).
2. **Категории:** пути категорий (`prodazha/kvartiry`, `arenda-dolgosrochnaya/...`,
   `prodazha/doma`, `prodazha/zemli` и т.д.).
3. **Схема фильтров `das[...]`:** параметры + enum-значения (тип дома, комнаты с
   синтаксисом `[]` и `N.100`, цена, год, продавец и пр.) — выверенная версия того,
   что сейчас в памяти.

## Сценарии использования (мои, как агента)

1. Пользователь просит спарсить по «регион X, район Y» → я беру готовый slug+гео-id из
   справочника и собираю URL за 0 запросов на разведку.
2. Нужен правильный синтаксис фильтра (комнаты N+, тип дома) → смотрю схему в справочнике.
3. Проверка валидности URL до парса — по справочнику, а не пробами на сайте.

## ЖЁСТКИЕ ОГРАНИЧЕНИЯ (главное в этой задаче)

- **Выполнять ТОЛЬКО после завершения текущего парса** (`besagash_phones.csv`).
  Никакой параллельной нагрузки на krisha.
- **Минимум запросов, строго последовательно, по-человечески** (паузы, как в парсере),
  через KZ-туннель. Цель — найти ЕДИНЫЙ авторитетный источник гео-дерева и забрать его
  за единицы запросов:
  - кандидаты: `robots.txt` → `sitemap.xml` (карта сайта со всеми гео-URL);
    JS-бандл с region data (`main-vendor`/`main-common`); ajax/JSON-эндпоинт автокомплита.
- **НЕ краулить по регионам** (это и есть запрещённое «дёрганье»). Если единого источника
  нет — сузить охват (см. границы), а не перебирать.
- Любой запрос к krisha по этой задаче — с явным логом «N/total», как договаривались.

## Формат и хранение (рекомендация на согласование)

- `docs/krisha-reference/geo.json` — гео-дерево (вложенное или плоский список с parent_id).
- `docs/krisha-reference/categories.json` — категории.
- `docs/krisha-reference/filters.json` — схема `das[...]` + enum'ы.
- `docs/krisha-reference/README.md` — как пользоваться + дата снимка + источник.
- Обновить память `tech_krisha_filters` ссылкой на этот справочник как на источник правды.
- Каждый файл со штампом даты снимка (данные krisha со временем меняются).

## Поддержка актуальности

- Снимок датируется. Ручное обновление по команде (скрипт `scripts/build-krisha-map.py`,
  который гентли тянет тот же единый источник). Авто-обновление — вне этой итерации.

## Границы (НЕ входит)

- Микрорайоны/улицы каждого города, если единый источник их не отдаёт дёшево
  (тогда — только до уровня, который даёт источник; не добивать краулингом).
- Цены/объявления/телефоны (это парсер, не справочник).
- Авто-рефреш по расписанию.
- Интеграция справочника в код парсера (валидация URL по справочнику) — отдельная итерация.

## Критерии готовности

- Источник найден и забран минимальным числом запросов (залогировано сколько).
- `geo.json` содержит хотя бы регионы + города + районы Алматинской обл. с slug+id,
  и Бесагаш (`besagash-dzerzhinskoe`) присутствует и корректен.
- `categories.json`, `filters.json` заполнены и сверены с памятью.
- README с датой снимка и инструкцией; память обновлена ссылкой.
- Ни одного «дифф-теста»/перебора по регионам в процессе сборки.
