# ТЗ: Браузерный поток показа телефона + обработка reCAPTCHA

**Дата:** 2026-06-07
**Статус:** на согласовании (Гейт 1)
**Связано с:** [2026-06-07-krisha-bot.md](2026-06-07-krisha-bot.md)

---

## Предпосылка (почему меняем)

krisha.kz ввёл **site-wide reCAPTCHA** на показ телефона (`/a/ajaxPhones`).
Подтверждено в сессии 2026-06-07: два независимых аккаунта (1 и 2) через KZ-туннель
(exit-IP Алматы/Kar-Tel) **залогинены и свободно браузят**, но **оба** получают
`gRecaptcha`-челлендж при показе телефона. Симптом не зависит от аккаунта → причина
site-wide, а не бан.

Текущий `requests`-поток (`bot/parser/krisha.py` → `ajaxPhones`) больше не может
получить телефон: он не способен пройти reCAPTCHA. Существующий `playwright_fb.py`
кликает кнопку и слушает ответ, но reCAPTCHA-челлендж **не обрабатывает**.

---

## Цель

Перевести показ телефона на **браузерный поток (Playwright)** как единственный путь,
с обработкой reCAPTCHA по схеме **browser-native сначала → 2captcha fallback**.
Сохранить имитацию человека (блокировки уже были) и интеграцию со scheduler/БД/MCP
без изменений снаружи.

---

## Решения Гейта 1 (зафиксировано с пользователем)

1. **reCAPTCHA:** browser-native сначала, **2captcha как fallback**.
2. **Solver-сервис:** **2captcha** (дешёвый, v2/v3, есть Python SDK).
3. **Глубина замены:** **всё через браузер** — и обход выдачи/пагинация, и показ
   телефона выполняются в одном Playwright-потоке. `requests`-парсинг выводится из
   игры для основного флоу.

---

## Сценарии использования

1. **Плановый прогон задания (24/7).** Scheduler запускает `run_task(task_id)`.
   Бот поднимает один браузер с куками аккаунта и KZ-прокси, проходит выдачу по URL
   задания, постранично собирает объявления, для каждого **нового** объявления
   открывает страницу, жмёт «показать телефон», получает номер (пройдя reCAPTCHA),
   сохраняет в БД. Человеческие паузы между действиями обязательны.

2. **reCAPTCHA проходится сама (happy path).** В реальном браузере с доверенной
   сессией + KZ-IP invisible/v3 reCAPTCHA выдаёт токен без видимого челленджа →
   телефон приходит сразу.

3. **reCAPTCHA требует решения (fallback).** Если появился видимый челлендж/`gRecaptcha`
   не прошёл — извлекаем sitekey, отправляем в 2captcha, получаем токен, внедряем его
   в страницу, повторяем показ телефона.

4. **Капча/ошибка не лечится → ротация аккаунта.** При устойчивом провале на текущем
   аккаунте — переключение на следующий (как сейчас в scheduler). Если на всех —
   объявление пропускается, лог.

5. **Ops/debug через CLI.** `python cli.py task run <id>` гоняет тот же браузерный
   поток для ручной проверки (1 объявление — без долбёжки).

## Архитектура

```
scheduler.run_task(task_id)
   └── parser.browser (НОВЫЙ основной поток, Playwright)
         ├── launch: Chromium + KZ-прокси (ALL_PROXY→socks5://) + cookies аккаунта
         ├── enumerate: обход выдачи + пагинация (паузы 2-5с)
         ├── для каждого нового объявления:
         │     ├── goto страницы объявления (пауза 1-3с)
         │     ├── click «показать телефон»
         │     ├── перехват ответа ajaxPhones
         │     └── recaptcha.handle:
         │            ├── browser-native (ждём авто-токен)
         │            └── 2captcha fallback (sitekey→token→inject→retry)
         └── save_phones → PostgreSQL (без изменений)
```

### Модули
- `bot/parser/browser.py` — **новый** основной браузерный поток (enumerate + reveal).
  Заменяет связку `krisha.py` + `playwright_fb.py` для основного флоу.
- `bot/parser/recaptcha.py` — **новый**: детект челленджа, browser-native ожидание,
  2captcha-fallback (извлечь sitekey, решить, внедрить токен).
- `bot/parser/human.py` — переиспользуем паузы; добавим браузерные жесты при необходимости.
- `bot/scheduler.py` — `run_task` переключается на `browser.py`; ротация аккаунтов
  сохраняется.
- `bot/secrets.py` — добавить чтение API-ключа 2captcha из Key Vault.
- `bot/parser/krisha.py`, `playwright_fb.py` — выводятся из основного флоу (оставить
  в репо как референс/легаси либо удалить — решим на Гейте 2).

### Секреты (Azure Key Vault — по правилу проекта)
- `krisha-bot--prod--TWOCAPTCHA-API-KEY` — ключ 2captcha. В рантайме через
  `DefaultAzureCredential`. В `.env` — только имя/URL KV, не сам ключ.

### Зависимости (пин версий, на подтверждение)
- `playwright` (уже в проекте) + установка Chromium.
- `2captcha-python` (SDK 2captcha) — **новая зависимость**, пин версии.

## Имитация человека (ОБЯЗАТЕЛЬНО — блокировки уже были)
- Паузы: между страницами выдачи 2–5с, между объявлениями 1–3с, перед кликом
  «показать телефон» — короткая пауза (имитация чтения).
- Один браузер/контекст на прогон, реалистичный UA, locale ru-RU; фингерпринт
  не меняется в рамках прогона.
- Снижение headless-детекта (navigator.webdriver и т.п.) — иначе reCAPTCHA-скоринг
  низкий и browser-native не пройдёт. Конкретику (stealth-аргументы / headful+xvfb)
  определим на Гейте 2.
- Никакой серийной долбёжки в тестах: максимум 1 объявление за тест, после капчи —
  пауза ≥5 мин.

---

## Границы (НЕ входит в эту итерацию)
- Ротация прокси/IP (один KZ-туннель остаётся как есть).
- Решение НЕ-reCAPTCHA челленджей (например, текстовых/hCaptcha).
- Headful+xvfb как обязательный режим (рассмотрим на Гейте 2, дефолт — headless+stealth).
- Изменения MCP-интерфейса и схемы БД (остаются прежними).
- Авто-пересохранение сессий (cookies истекают — отдельная задача).

---

## Критерии готовности
- `python cli.py task run <id>` через KZ-прокси на **одном новом** объявлении
  возвращает телефон (browser-native ИЛИ через 2captcha), без видимого падения на
  reCAPTCHA.
- Телефон сохраняется в БД, дедупликация работает.
- Соблюдены человеческие паузы; нет серийных запросов без задержек.
- Ротация аккаунта срабатывает при устойчивом провале.
- Секрет 2captcha читается из Key Vault, не хранится в коде/.env.
